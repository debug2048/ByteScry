# ByteScry Design Notes

ByteScry is a pluggable Java and Android decompiler workbench. The main product
surface is the JavaFX GUI, with a CLI kept for scripted Java class and JAR
workflows.

The repository uses `bytescry-*` module names and `com.github.bytescry` Java
packages.

## Goals

- Make Java and Android artifact inspection usable without asking the user to
  choose low-level strategies first.
- Prefer high-quality external engines for source recovery.
- Keep a lightweight internal engine as a fallback and as a readable baseline.
- Keep engine failures visible but non-blocking whenever partial output is
  usable.
- Optimize the GUI for repeated analysis: quick project loading feedback,
  searchable source, side-by-side compare, export, and diagnostics.

## Module Layout

| Module | Responsibility |
| --- | --- |
| `bytescry-core` | Engine interfaces, engine registry, class/artifact loading, bytecode rendering, CFR, Vineflower, JADX, Simple |
| `bytescry-cli` | picocli wrapper for Java `.class`, `.jar`, and directory workflows |
| `bytescry-gui` | JavaFX desktop application |
| `bytescry-tests` | Versioned Java sample tests |
| `repo` | Project-local CFR artifact |

## Core Abstractions

### `ClassFile`

Represents a class-like item in the workspace.

- `className`: internal name such as `com/example/App`
- `bytes`: JVM class bytes when available
- `source`: original file or artifact path

For Android source entries generated by JADX, `bytes` can be empty because the
GUI is browsing generated Java source rather than JVM bytecode.

### `DecompilerEngine`

Common engine contract:

- `decompile(ClassFile, DecompilerOptions)`
- `decompile(List<ClassFile>, DecompilerOptions)`
- `getName()`

The registry currently registers:

- `cfr`
- `vineflower`
- `jadx`
- `simple`

### `ArtifactDecompilerEngine`

Optional capability for engines that operate on whole artifacts:

- `supportsArtifact(Path)`
- `exportArtifact(Path, Path, DecompilerOptions)`
- `loadArtifactClasses(Path, DecompilerOptions)`

`JadxEngine` implements this for Android inputs. The GUI uses
`loadArtifactClasses` to populate Project with generated source classes instead
of showing only a single APK placeholder.

### `DecompilationResult`

Carries:

- class name
- source code
- engine name
- elapsed time
- error when decompilation failed

The GUI caches successful and failed results per engine/class to avoid
unnecessary repeated work.

## Engine Strategy

### CFR

CFR is the default Java decompiler for `.class`, `.jar`, and class directories.
The implementation invokes CFR in a separate Java process and reads generated
`.java` files from a temporary output directory.

The process boundary prevents engine-side exits or global state from disrupting
the GUI.

### Vineflower

Vineflower is available as an alternate Java decompiler. It is used for Compare
and can be selected manually in Source and Export for Java inputs.

It also runs as a Java engine over temporary inputs/outputs so the GUI remains
isolated from engine-specific file handling.

### JADX

JADX is the Android engine for `.apk`, `.dex`, `.aab`, `.apks`, `.apkm`, and
`.xapk`.

Important behavior:

- Android inputs show only `jadx` in the GUI engine dropdown.
- The GUI uses JADX to decompile the whole artifact and index all generated
  `.java` files into Project.
- Export writes the whole JADX output.
- Nonzero JADX exit codes are not always fatal. If `.java` files were produced,
  the result is treated as usable and partial errors are surfaced through status
  or diagnostics.

### Simple

Simple is an internal lightweight engine built around ASM. It is not intended to
match CFR or Vineflower quality. It is useful for:

- fallback output when a high-quality engine fails
- quick structural inspection
- tests around bytecode and type utilities

## GUI Design

### Window

The JavaFX stage is undecorated but implements native-like controls:

- minimize
- maximize/restore
- close
- double-click maximize
- drag-to-move
- drag from maximized state to restore and move

Startup size and position are calculated from the current screen
`visualBounds`, which avoids placing the title/menu row outside visible screen
space on small or high-DPI laptop displays.

### Project

Project is the workspace tree.

- New loads clear the old tree immediately.
- Loading disables the tree and search box so stale classes cannot be selected.
- A Project-level spinner shows loading state.
- Long paths use ellipsis in the label and full-path tooltip on hover.
- Packages are displayed in collapsed form such as `com.example.service`.
- Right-click copies package names and class names.

### Source

Source is the primary view.

- Engine list adapts to the loaded input type.
- Java inputs: `cfr`, `vineflower`, `simple`.
- Android inputs: `jadx`.
- Line numbers are rendered in a nonselectable gutter.
- Search highlights and cycles through matches.

### Compare

Compare is side-by-side.

- Java: CFR vs Vineflower, or CFR vs Simple when Vineflower is unavailable.
- Android: JADX on the left; bytecode comparison is disabled on the right.
- Search spans both panes and reports match position in the status bar.

### Bytecode

Bytecode uses ASM for JVM `.class` inputs. Runtime ASM failures are caught and
shown as a readable message instead of crashing the JavaFX thread.

Android artifacts do not use this view because APK/DEX content is not JVM
`.class` bytecode.

### Diagnostics

Diagnostics is optional from the View menu. It records failures and fallbacks
from decompilation, compare, bytecode, and export operations.

### Export

Export uses a custom styled dialog so it can follow the application theme.

- User chooses output directory.
- User chooses engine.
- Android artifacts route to whole-artifact JADX export.
- Java exports can fall back to Simple per class when selected high-quality
  engines fail.

## Loading Rules

### File

- `.class`: read bytecode and class name with ASM when needed.
- `.jar`: enumerate `.class` entries.
- Android artifacts: use JADX whole-artifact indexing in the GUI.

### Directory

- Recursively load `.class` files.
- Also read direct child `.jar` and Android artifact files.
- If no `.class` files are found and exactly one direct child artifact exists,
  open that artifact directly.

## Accuracy Model

ByteScry cannot guarantee exact original source. The best possible output still
depends on:

- compiler transformations
- missing debug metadata
- obfuscation
- synthetic bridge/accessor methods
- lambdas and invokedynamic rewrites
- Android resource and multidex layout
- malformed or protected bytecode

The UX is designed around this reality: show the best source by default, allow
engine switching, compare alternatives, keep diagnostics visible, and export
best-effort results.

## Testing

Current tests cover:

- class file and artifact loading behavior
- Simple engine output for representative Java constructs
- type utilities
- Java 7/8/11/17 sample decompilation integration tests

Recommended future GUI tests:

- open `.jar` and verify Project clears stale tree while loading
- open Android artifact and verify JADX-generated Project entries
- search behavior in Source and Compare
- export dialog validation
